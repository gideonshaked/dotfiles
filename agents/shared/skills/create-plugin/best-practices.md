# Claude Code Plugin Best Practices

## Core Rules

- Skills run inside Claude Code's conversation. All script output goes through the Bash tool and is read by Claude, not rendered in a terminal. No ANSI, no rich, no interactive TUI.
- Do not use fenced code blocks (triple backticks) in SKILL.md: Claude Code's preprocessor treats them as shell injection (! syntax). Write commands as plain text that Claude executes via the Bash tool.
- ${CLAUDE_SKILL_DIR} is substituted in skill content. Shell variables ($RUN, $PROJECT) defined in skill content are NOT expanded by the preprocessor; they only work if Claude runs them in a Bash tool call.
- Plugin .mcp.json servers conflict with user-configured servers of the same name/URL, showing as "Plugin Errors." Use a SessionStart hook to check and install MCPs only if missing, instead of declaring them in .mcp.json.
- userConfig fields require title (string) and type (one of string/number/boolean/directory/file). Missing these causes manifest validation failure. Only use userConfig for values that genuinely need user input at install time.
- --plugin-dir copies the plugin at session start. Mid-session edits require a full restart, not just /reload-plugins.
- All non-user files in plugin output should be dotfiles/dotfolders. User-facing output (reports, PDFs) should be topically named.
- Prefer deterministic Python code over Claude for structured output (BibTeX, JSON schemas). Use Claude for synthesis and analysis, code for formatting and data transformation.
- When spawning claude -p subprocesses, use --allowedTools to restrict each agent's tool access. Agents should be leaf-only; orchestration and parallelism belong in the Python code via asyncio.gather.
- Use Jinja2 templates (.j2 files) for prompts sent to claude -p. Use tojson filter for structured data. Require "respond with ONLY a JSON code block, no prose" for any prompt expecting parseable output.

## Directory Structure and Packaging

- Use skills/<name>/SKILL.md (not commands/<name>.md) -- this is the modern format. The skills/ directory is guaranteed to be cached when the plugin is installed from a marketplace.
- Put all Python code in bin/, this directory is guaranteed to be on PATH and cached. Create a single shell entrypoint (e.g. bin/peer-review-cli) that calls uv run --project "$CLAUDE_PLUGIN_ROOT" python -m bin.cli, with bin/cli.py dispatching to subcommands. This means SKILL.md uses short commands (my-cli do-thing) instead of long uv run --project paths.
- Do NOT put code in a custom scripts/ directory -- it is not a recognized plugin directory and may not survive caching when installed from a marketplace.
- Keep templates (.j2 files) and other code resources in bin/ alongside the Python files. Keep content resources (check definitions, reference docs) in the skills/<name>/ directory alongside SKILL.md.
- Use uv run --project (not --directory) for module resolution. --directory changes CWD which breaks module resolution in cached plugin locations. --project discovers the pyproject.toml without changing CWD.
- marketplace.json source paths must start with ./ (not just .) -- e.g. "source": "./" for a plugin at the repo root.
- Use ${CLAUDE_PLUGIN_ROOT} in bin/ scripts to find the project root. It's available as an env var in plugin subprocesses. Fall back to $(dirname "$0")/.. for local development.
- Version in plugin.json must be bumped for marketplace updates to be detected. If you don't bump the version, users won't see changes.

## SKILL.md Architecture

- SKILL.md should be an orchestrator, not a logic file. Any deterministic steps (validation, state reconciliation, file writing) belong in tested scripts that Claude calls and reads output from. Claude's role is judge/orchestrator: making decisions where indeterminism exists.
- Keep SKILL.md as light as possible. If a step exceeds a paragraph, move its description into a separate .md file in a references/ directory. The SKILL.md step becomes "read this file" with a short summary. This prevents any one step from dominating context.
- Each SKILL.md step should either: (1) be simple enough to describe inline, (2) call a script and read its output, or (3) tell Claude to read a reference file for full instructions.

## Output and Formatting

- All user-facing formatted output (settings displays, progress headers, summaries) should be generated by deterministic scripts, not by Claude. Claude's formatting varies between runs; scripts produce consistent output.
- SKILL.md should include an output visibility instruction: tell Claude to print script output as direct text responses, not leave it collapsed inside Bash tool results. Without this, long outputs get hidden behind "ctrl+o to expand."
- Use the YAML frontmatter fully: name, description (front-load the key use case), argument-hint, allowed-tools (as YAML list), effort (high for complex multi-step skills). Use effort: high for skills that run loops or spawn subprocesses.
- The description field in frontmatter drives when Claude auto-triggers the skill. Include trigger phrases like "TRIGGER when: user asks for X, says Y, does Z."
- For formatted output scripts, box-drawing characters render well in Claude Code's monospace output and look more polished than markdown.

## State Management and Recovery

- Manage plugin state via the filesystem, not Claude's memory. Write state files at key checkpoints so sessions can recover from interruptions.
- Use a setup/validation script at session start that reconciles persisted state with the current environment. This gives deterministic recovery rather than hoping Claude remembers where it was.
- If using Claude Code Tasks, write task files directly to the tasks directory from a script rather than having Claude chain dozens of TaskCreate/TaskUpdate calls. One TaskList call after script-based reconciliation is more reliable than 40+ chained tool calls.
- Consider recommending /clear at key workflow milestones to manage context. As long as recovery from state files is solid, this keeps Claude sharp for the next phase.

## Context Management

- Context is finite and degrades with size. Validate early in the session so failures happen before context fills up. Use a setup script to check environment, dependencies, and state before real work begins.
- Use subagents for self-contained units of work that don't need user input. Each subagent gets a fresh context window divorced from the main session. This is primarily a context management tool, not just a parallelism tool.
- For long-running plugins, bring the user in at key moments via AskUserQuestion rather than letting Claude make assumptions and churn tokens. Automation should increase over time, but user checkpoints prevent expensive wrong turns.

## Hooks

- SessionStart hook fires on startup and after /clear. Use it to write session-specific env vars or bootstrap state. Note: CLAUDE_ENV_FILE may not persist after /clear; if you need the session ID post-clear, have the hook output JSON that Claude passes to setup scripts.
- SubagentStop hook fires when a subagent completes and can access the subagent's .jsonl interaction log. Use this to extract subagent output and write files without giving subagents blanket Write permissions. Validate .jsonl schema assumptions in setup scripts since the format may change.
